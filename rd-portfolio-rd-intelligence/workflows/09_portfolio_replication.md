# Workflow 09 — Portfolio Replication

## Purpose

Make the workflow reusable across product platforms, material systems, business units, and geographies.

## Replication Logic

When transferring learning from one portfolio to another, classify each item as:

```text
Universal workflow logic — can be reused directly
Technical relationship — must be validated in target material system
Test method — may require method harmonization
Design rule — requires context and approval
Regulatory claim — region/product-specific
Manufacturing rule — site/process-specific
Supplier assumption — portfolio and region-specific
```

## Transferability Assessment

Use this matrix:

```markdown
| Learning / Template | Source Portfolio | Target Portfolio | Transferability | What Must Be Revalidated | Owner |
|---|---|---|---|---|---|
```

## Portfolio Playbook Output

```markdown
# Portfolio Replication Playbook

## Reusable Across All Portfolios
- VOC intake structure
- CTQ matrix structure
- Evidence scoring rubric
- Patent extraction template
- X-Y synthesis template
- Post-experiment learning record

## Requires Portfolio Customization
- CTQ targets
- Test methods
- competitor list
- regulatory claims
- manufacturing constraints
- approved material library
- supplier qualification requirements

## Requires Platform Validation
- X-Y relationships
- design rules
- process windows
- aging / sterilization assumptions
- performance tradeoffs

## Portfolio-Specific Setup Checklist
1. Define product family.
2. Define platform / material system.
3. Define target regions.
4. Load approved test methods.
5. Load competitor and benchmark list.
6. Load internal material library.
7. Load manufacturing constraints.
8. Define review owners.
9. Define data governance rules.
10. Run pilot project.
```

## Quality Check

Do not mark a technical learning as universal unless it has been validated across relevant platforms or explicitly approved as a general design principle.
