# VOC Scout 交互规范

## 0. 执行清单

每次处理 VOC 时，以下 11 步作为硬约束，不可跳过：

```
VOC Scout 阶段:
□ 1. 领域分类（材料/化工? 电子? 医疗? 机械?）
        → 选择偏重的搜索数据库和 IPC 大类
□ 2. 市占率搜索（中英双语，先于专利搜索）
        → "行业名 + global market share + top suppliers"（英文）
        → "行业名 + 市场规模 + 竞争格局 + 龙头企业"（中文）
        → 输出: 按市占率排序的公司列表（带份额%）
        → ⚠️ 专利数≠市占率，不可用专利检索量倒推龙头
□ 3. 识别龙头（LLM 领域知识 + 市占率数据交叉验证）
□ 4. Round 1 全景展开 → 用户多选（按市占率排序，P0标注份额）
□ 5. Round 2 聚焦深入 → 用户排除/选择
        → 可选 Round 3 细化维度
□ 6. 输出三件套 JSON
        → 🏢 公司 + 🔑 关键词 + 📂 IPC + ⚠️ FTO

Deep Search 阶段:
□ 7. 第一路搜索: 公司定向（每家 P0/P1 公司 2 轮搜索）
□ 8. 第二路搜索: 关键词 + IPC 全量扫荡（不限公司）
□ 9. 提取新词: 新关键词 / 新公司名 / 新 IPC / 新路线
        → 用 §5.7.4 的提取模板
□ 10. 第三路搜索: 用新词做下一轮迭代搜索
□ 11. 收敛判断: 检查 §5.6.1 的 3 项条件 + 定量指标
□ 12. 输出完整专利全景（含 CTQ 对照表 + FTO 评估）输出三件套 JSON
        → 🏢 公司 + 🔑 关键词 + 📂 IPC + ⚠️ FTO
```

## 1. 概述

### 1.1 定位

VOC Scout 是一个交互式探索模块。输入是简短粗糙的客户需求（VOC），输出是为 **Deep Search（深度检索）** 做好准备的三件套：

```
🏢 主要公司 + 技术方向
🔑 专利搜索关键词（已收敛验证）
📂 IPC 分类范围
⚠️ FTO 注意事项
```

### 1.2 边界

**VOC Scout 做:**
- 2-3 轮交互收敛用户需求
- 识别行业内主流公司和主流技术路线
- 生成验证过的搜索关键词和 IPC
- 标注已知的 FTO 风险（过期/有效/审查中专利）

**VOC Scout 不做:**
- ✗ 列出 Top N 专利列表（那是 Deep Search 的事）
- ✗ CTQ 指标定值
- ✗ 测试方案建议
- ✗ 配方/材料推荐
- ✗ 深入的权利要求分析
- ✗ 详尽的 FTO 分析（Deep Search 阶段做）

### 1.3 核心原则

```
1. 先列龙头，再搜专利
   不靠随机关键词碰运气，先用领域知识确定行业主要玩家
   龙头识别要区分: 专利龙头 vs 产品市占龙头 vs 技术原创龙头

2. 用户做选择题，系统做搜索
   每轮展示 4-6 个选项，用户多选，系统根据选择深入下一轮

3. 过期专利也是宝
   过期专利无 FTO 风险，技术公开可自由使用，反而更有价值

4. 三路搜索防遗漏
   一路: 公司定向（找标杆，含产品手册）
   二路: 关键词+IPC 扫荡（防漏小公司有效专利）
   三路: 新词提取→迭代（搜索反哺搜索）

5. 搜索范围不限于专利
   专利 + 产品 TDS/MSDS + 学术论文 + 临床注册数据 + 行业白皮书
   实战中 50% 以上的关键参数来自非专利来源

6. 诚实 > 全能
   不确定时标注信心度，不熟悉的领域承认局限性
   不会就是不会，不假装自己懂
```

### 1.4 领域预分类

在 Round 1 之前，必须先做领域分类。不同领域使用不同的搜索策略：

| 领域 | 主要 IPC | 搜索重点 | 额外数据源 |
|------|---------|---------|-----------|
| **材料/化工/高分子** | C08, C09, B32B | 专利 + 产品 TDS | — |
| **电子/半导体** | H01, H05, G02 | 专利 + 行业白皮书 | 显示行业报告 |
| **医疗/器械** | A61, A61L, A61F | 专利 + 临床数据 | PubMed, FDA PMA |
| **机械/结构** | B…, F16 | 专利 + 产品规格书 | — |
| **法规驱动型**（汽车/食品接触） | 各类 | 专利 + 法规文件 + 测试数据 | VDA, FDA, REACH |

### 1.5 信心度标注

每轮输出必须标注系统对该轮内容来源的信心度：

| 标注 | 含义 | 说明 |
|------|------|------|
| `★★★` 高信心 | 基于明确专利数据+产品资料 | 数值和公司名经过双重验证 |
| `★★☆` 中信心 | 基于 LLM 领域知识+部分搜索结果 | 龙头判断基本正确，细节待验证 |
| `★☆☆` 低信心 | 首次接触该领域 | 建议找领域专家验证，系统可能漏判断格局 |

**规则：**
- Round 1 首次接触新领域时默认标 `★★☆`
- 如果用户纠正 ≥2 次，自动降到 `★☆☆`
- 如果三轮 Deep Search 后格局稳定且无矛盾，升到 `★★★`

## 2. 用户交互流程

### 2.1 流程总图

```
用户输入 VOC（一句话/一段话）
         │
         ▼
   ┌─ 领域预分类 + 信心度标注 ─┐
         │
         ▼
┌─────────────────────────────┐
│  Round 1: 全景展开           │
│  系统展示 4-6 条技术路线/方向  │
│  用户选 N 个（多选）          │
└─────────────┬───────────────┘
         │
         ▼
┌─────────────────────────────┐
│  Round 2: 聚焦深入           │
│  展示各方向下的主要公司       │
│  + 核心技术 + 专利状态       │
│  用户排除不相关/选择想深挖的  │
└─────────────┬───────────────┘
         │
    ┌────┴────┐
    │         │
  还不够    够了
    │         │
    ▼         ▼
┌────────┐  ┌─────────────────────────────┐
│Round 3 │  │  输出三件套                  │
│细化维度 │  │  🏢 公司 + 🔑 关键词         │
└──┬─────┘  │  📂 IPC + ⚠️ FTO            │
   │        │  ★★☆ 信心度                 │
   └───────→│  [导出JSON] [开始深度搜索]    │
            └─────────────────────────────┘
```

### 2.2 Round 1 — 全景展开

**目标：** 让用户快速看到这个领域有哪些技术方向。

**展示内容（4-6 个选项）：**

每条选项包含：
- 方向名称（公司名 + 技术特征）
- 一句话说明核心技术特征
- 代表性公司/产品
- 差异化特征（固含量、专利状态等）

**格式示例：**

```
□ 1. BASF 路线 — 高固含通用型
    Acronal V 215 量产标杆，69%固含，剥离力高
    核心专利已过期，无FTO风险

□ 2. Avery Dennison 路线 — 多峰粒径+高速涂布
    bimodal 粒径设计，≥68%固含，600-800m/min 涂布
```

**用户操作：** 多选勾选。支持选 1 个到选全部。

**规则：**
- 选项 4-6 个，尽量正交，但如果行业重叠则如实呈现
- 不预设用户的知识水平—小白看得懂、专家快速扫读
- 如 VOC 中存在明显矛盾，在选项旁边标注

### 2.3 Round 2 — 聚焦深入

**目标：** 将用户选的方向展开。

**展示内容：**

每个方向展开为：
- 主要公司
- 核心专利号 + 当前状态（有效/过期/审查中）
- 关键技术指标数值
- 差异化优势
- 注意事项

**优先级排序格式：**

```
P0 ★ BASF      对标 Acronal V 215（69%固含），核心专利已过期
P1 ★ Avery     多峰粒径技术，适合高速量产
P2 ★ 项立新材料 种子返滴法，国内参考
```

**用户操作：**
- 进一步细化（如"看 BASF 配方方向"）
- 排除已选分支
- 可选"够了，直接输出"

### 2.4 Round 3（可选）— 进一步细化

触发条件：
1. Round 2 后用户还有多个方向不确定
2. 用户主动选"继续细化"
3. 需要区分应用场景（线束用 / 面板固定用 / 泡棉复合用）

**内容：** 更细的维度对比或应用场景细分。

### 2.5 输出三件套

```
┌──────────────────────────────────────────────┐
│  🏢 目标公司（按优先级排序）                    │
│  P0 ★ ...      [T] 产品TDS确认                │
│  P1 ★ ...      [P] 专利确认                   │
│  国内参考 ...   [A] 论文 + [D] 公开资料         │
├──────────────────────────────────────────────┤
│  🔑 搜索关键词                                 │
│  核心: ...    补充: ...    排除: ...           │
├──────────────────────────────────────────────┤
│  📂 IPC 分类                                   │
│  核心: ...    补充: ...                        │
├──────────────────────────────────────────────┤
│  ⚠️ FTO 注意事项                                │
│  各公司核心专利状态 + FTO 扫荡方向              │
│  已知规避思路                                   │
├──────────────────────────────────────────────┤
│  ★★☆ 本报告信心度                               │
│  📋 对话路径（N 轮选择记录）                    │
│  [导出JSON] [重新调整] [开始深度搜索]           │
└──────────────────────────────────────────────┘
```

### 2.6 Session State 管理

VOC Scout + Deep Search 流程跨 5+ API 调用，需要携带中间状态。采用**前端维护，每次回传**的最小方案：

```
不引入数据库/Redis。前端在 React state 中维护 session 对象，
每次 API 调用时将其作为请求的一部分传回后端。
```

**Session 对象结构：**

```json
{
  "session_id": "uuid-generated-by-frontend",
  "voc": "原始VOC文本",
  "domain_class": "材料/化工",
  "phase": "scout_round2",
  "scout": {
    "round1": {
      "routes": [...],
      "selected_ids": ["1", "3"]
    },
    "round2": {
      "routes": [...]
    },
    "output": {
      "companies": {...},
      "keywords": {...},
      "ipc": {...},
      "fto": {...}
    }
  },
  "deep_search": {
    "round_results": [...],
    "convergence": {...}
  },
  "corrected_by_user": [
    {"phase": "round1", "correction": "3M还有微孔涂布技术"},
    {"phase": "deep_search_r1", "correction": "KCI是NPWT, 排除"}
  ],
  "confidence": "★★☆",
  "created_at": "2026-06-22T10:00:00Z"
}
```

**API 调用传参约定：**

```
POST /api/voc-scout/round1
  Request:  {"voc": "...", "session_id": "uuid"}
  Response: {"routes": [...], "session_id": "uuid"}

POST /api/voc-scout/round2
  Request:  {"session_id": "uuid", "selected_route_ids": ["1","3"]}
  Response: {"routes": [...], "session_id": "uuid"}

POST /api/voc-scout/output
  Request:  {"session_id": "uuid"}
  Response: {"companies": {...}, "keywords": {...}, ...}

POST /api/deep-search/start
  Request:  {"session_id": "uuid"}
  → 后端从 session 中读取 Scout output 作为输入

POST /api/deep-search/map-reduce
  Request:  {"session_id": "uuid", "patent_numbers": [...]}
  → 后端追加结果到 session
```

**前端状态机：**

```
'input' → 'scout_round1' → 'scout_round2' → 'scout_output'
     → 'deep_search_start' → 'deep_search_map_reduce' → 'deep_search_result'
```

**会话持久化（可选，v2）：** 可将 session JSON 写入 `.claude/sessions/{session_id}.json`，支持断点续传和对话回溯。

## 3. 关键技术原则

### 3.1 公司识别策略

先确定市占率排名，再做专利搜索。不可用专利检索量倒推市场地位。

**Step 0（必须在专利搜索之前）: 市占率搜索**

```
用中英双语搜索全球市场份额排名:

  英文: "<industry name>" + "global market share" + "top suppliers" + "competitive landscape"
  中文: "<行业名>" + "市场规模" + "竞争格局" + "龙头企业"

  输出: 按市占率排序的 Top 10 公司列表（带份额%）

⚠️ 绝对禁止: 用专利搜索结果数量倒推市场地位
   → 专利最多的公司不一定是市占龙头
   → 中文环境运营的公司可能在英文专利搜索中不可见
   → 实战教训: 汽车座椅电机领域，胜华波(19.22%)专利主要为中文 CN，英文搜索几乎不出现
```

**Step 1: 区分三种"龙头"**

```
这个领域全球最大的 3-5 家公司是谁？（市占龙头 → 来自 Step 0 市占率数据）
他们的核心技术是谁发明的？（技术原创龙头）
他们的核心专利还在不在有效期？（FTO 龙头 — 过期无风险优先）
```

| 类型 | 示例 | 标志 | 来源 |
|------|------|------|------|
| **市占龙头** | 胜华波（19.22%, 座椅电机） | 商业化产品量最大 | `[D]` 行业报告 |
| **专利龙头** | tesa（双峰PSA一家独大） | 核心专利覆盖面广 | `[P]` |
| **技术原创龙头** | Nidec（无槽BLDC原理首创） | 基础专利 | `[P]` |

**⚠️ 三者可能重叠，也可能完全不同。不可假设一致。**

**中英双语搜索是硬约束：**

```
对每个领域，必须同时执行:
  英文: "company name" + "technology keyword" + "patent"
  中文: "公司名" + "技术关键词" + "专利"

  原因: 很多全球龙头在中国以中文名运营，英文搜索可能完全找不到
  - Leggett & Platt → 礼恩派（中国三个基地，英文搜索几乎不出现）
  - Shenghuabo → 胜华波（全球第一，但音译名在英文中不可见）
  - ASMO → 阿斯莫（Denso子公司，日文/中文专利体系）

  也搜产品资料: TDS/MSDS/产品手册/白皮书（中英双语）
```

### 3.2 来源可信度标注

每个 CTQ 数据点必须标注来源类型：

| 标注 | 来源类型 | 可信度 | 示例 |
|------|---------|--------|------|
| `[P]` | 专利数据 | 最高（有法律约束） | "溶胀率<30% vol (US10717890B2 Example 3)" |
| `[T]` | 产品 TDS/白皮书 | 高（商业化验证） | "Acronal V 215 固含 69% (BASF 产品手册)" |
| `[A]` | 学术论文 | 中高（经同行评审） | "Solef 5130 吸收率 11.3% (Oxford Open Mat Sci 2023)" |
| `[C]` | 临床数据/FDA注册 | 高（有监管背书） | "Orsiro 支架梁厚 60μm (FDA PMA P170030)" |
| `[D]` | 领域知识/行业报道 | 中（需验证） | "Kolon 是三星 Galaxy Fold 的 CPI 供应商" |
| `[E]` | 估算/推理 | 低（仅供参考） | "估计到期日~2030（假设标准20年期限）" |

**规则：**
- 优先引用 `[P]` `[T]` `[C]` 来源
- `[E]` 不得独立作为推荐依据
- 输出时按来源可信度排序，`[P]` `[T]` `[C]` 在前

### 3.3 专利状态标注规则

| 状态 | 标注 | 说明 |
|------|------|------|
| 已过期 | `✅ 已过期` | 技术可自由使用 |
| ~2030到期 | `⚠️ ~2030到期` | 标注到期年份，加 6 个月安全 margin |
| 审查中 | `⏳ 审查中` | 需跟踪授权范围 |
| 状态不明 | `❓ 需核查` | 标注建议核查方向 |

**专利到期估算规则：**

```
中国发明专利: 申请日 + 20年
美国专利:     申请日 + 20年（1995.6.8 后）/ 授权日 + 17年（之前）
欧洲专利:     申请日 + 20年
日本专利:     申请日 + 20年
实用新型（中/日/韩/德）: 申请日 + 10年

⚠️ 注意:
  - 到期日可能受 PTA（Patent Term Adjustment）/ PTE（Patent Term Extension）影响
  - 标注时加安全边界: 标注到期日 + 6个月
  - 药品/医疗器械可能有额外独占期（regulatory exclusivity）
  - 不确认时默认标注 "❓ 需核查"
```

### 3.4 选项设计规则

- 每轮 4-6 个选项
- 选项尽量正交，但行业重叠时如实呈现
- 选项描述含**公司名 + 技术特征 + 一句话关键信息**
- 不使用模糊标签（"方案A/B"），使用具体公司和产品

### 3.5 交互节奏控制

- 默认 2 轮收敛，最多 3 轮
- 每轮提供"够了，直接输出"的出口
- 每轮支持"回退到上一轮"
- 用户的每轮选择记录在对话路径中

### 3.6 矛盾/张力标注

R&D VOC 中常见的四类矛盾，Round 1/2 时应主动识别：

| 类型 | 示例 | 标注方式 |
|------|------|---------|
| **性能矛盾** | "高固含+低粘度"（选工艺路线但要求高粘性） | ⚡ 技术矛盾 |
| **应用错配** | 搜到NPWT密封敷料却把它放在"高透气"路线 | 🔀 应用归类错误 |
| **合规冲突** | "VDA 278达标" ↔ "高速涂布下溶剂残留高" | 🏛 合规张力 |
| **成本矛盾** | 技术路线性能好但原料/工艺成本远超目标 | 💰 成本可行性 |

标注方式：

```
⚡ 注意：你选了 A+B+C 三个工艺方向，但目标应用是 X，
    其中 C 路线的主流方案不太匹配 X 应用。
    建议在 Deep Search 时优先关注 A 和 B。
```

## 4. Deep Search 的已知局限与诚实声明

### 4.0 不可能做到"全量检索"

用户问"你能确保这家公司的所有相关专利都检索到了吗"——答案是**不能**。这是系统级的固有限制，不是执行问题。

#### 4.0.1 五大不可消除的盲区

| 盲区 | 原因 | 影响 | 缓解措施 |
|------|------|------|---------|
| **语言壁垒** | 日文(JP特開)、德文(DE)、韩文(KR)专利在英文搜索中不可见 | 日本/德国/韩国公司的核心专利可能大量遗漏 | 对日/德/韩公司，手动加本地语言关键词搜索 |
| **公司名变体** | L&P = Leggett & Platt = 礼恩派 = 常熟礼恩派汽车动力 = 无锡礼恩派华光 | 同一家公司以 5 个名字分散在不同专利局 | 搜完英文名后，补搜中文/本地名 |
| **子公司/收购/剥离** | ASMO→Denso, Solventum←3M, 多层级子公司 | 专利分散在多个 assignee 名下 | 搜母公司名 + 子公司名，双向验证 |
| **数据库局限** | Google Patents 不索引全部中国实用新型；Espacenet 日文摘要翻译不全；WebSearch 非结构化 | 每个工具都有固定盲区 | **代码实现时必须用 patent_search.py 抓取全文分析，不得依赖 AI WebSearch 摘要或 LLM 合成数据** |
| **贴牌/OEM制造** | 大量中国厂商不申请专利，靠 trade secret 或贴牌生产 | 实际产能最大的公司可能在专利库中完全不可见 | 市占率数据补盲（§3.1 Step 0） |

#### 4.0.2 实战数据：不同领域的估计覆盖率

| 领域 | 估计覆盖率 | 主要盲区 |
|------|-----------|---------|
| 材料/化工（BASF, 3M, Arkema...） | **60-75%** | 美欧日专利体系较完整 |
| 医疗（3M, S&N, Zhende...） | **50-65%** | 中国实用新型、FDA PMA 数据 |
| 机械/机电（Brose, 胜华波, Mabuchi...） | **30-50%** | 日文/德文/韩文专利壁垒严重 |
| 电子（Samsung, LG, BOE...） | **50-70%** | 韩文/中文专利部分遗漏 |

#### 4.0.3 最危险的遗漏类型

```
⚠️ 致命遗漏（可能导致FTO漏检）:
  1. 日本公司的日文专利 → Mabuchi 在日文专利库中的座椅电机专利
  2. 德国公司的德文专利 → Brose 在 DE 的早期基础专利
  3. 中国实用新型 → 授权快、公开快，但 Google Patents 索引不全
  4. 子公司的专利 → 母公司名下搜不到

⚠️ 中度遗漏（影响竞争分析完整性）:
  5. 韩国公司的韩文专利 → Keyang 的韩文专利
  6. 同族专利不同国家版本 → 同一发明在不同国家的法律状态不同
  7. 商标/品牌名与技术名不匹配 → 产品叫 Acronal，专利 assignee 叫 BASF

⚠️ 轻度遗漏（不影响核心判断）:
  8. 学术论文中的未申请专利的技术方案
  9. 过期 20 年以上的基础专利（不影响 FTO）
```

#### 4.0.4 诚实声明模板

  - 日文/德文/韩文专利未系统检索

## 5. Deep Search（深度检索）— VOC Scout 的下游环节

### 4.1 定位

VOC Scout 输出 🏢 公司 + 🔑 关键词 + 📂 IPC + ⚠️ FTO 四件套后，进入 Deep Search 阶段：

```
VOC Scout 输出                    Deep Search 执行
┌─────────────────┐              ┌──────────────────────────┐
│  🏢 目标公司      │ ──→ │ 按公司定向搜（专利+产品）   │
│  🔑 搜索关键词    │ ──→ │ 按关键词+IPC全量扫荡       │
│  📂 IPC 分类      │ ──→ │ 按分类号批量检索           │
│  ⚠️ FTO 注意      │ ──→ │ 核查关键专利权利要求       │
│  ★★☆ 信心度      │ ──→ │ 标注来源可信度             │
└─────────────────┘              └──────────┬───────────────┘
                                            ↓
                           ┌──────────────────────────────────┐
                           │  Deep Search 输出                 │
                           │                                  │
                           │  ① 完整专利全景                  │
                           │  ② CTQ 参数对照表（含来源标注）   │
                           │  ③ 技术路线分类 + 发展世代        │
                           │  ④ FTO 风险评估                  │
                           │  ⑤ 推荐策略（短期/中期/长期）     │
                           └──────────────────────────────────┘
```

### 4.2 Deep Search 的三轮迭代搜索法

通过 6+ 个跨领域 VOC 实战验证的核心方法论。

#### 4.2.1 迭代流程

```
Round 1 — 初始搜索
  用 VOC 原文 + 领域已知龙头做广泛搜索
  搜索范围: 专利 + 产品资料 + 学术论文
  目的: 快速建立领域认知，发现主要玩家和技术路线
  每路搜索做: 公司定向 + 关键词扫荡 + 新词提取

Round 2 — 用新关键词再搜
  从 Round 1 结果中提取"新出现的术语/公司/IPC"
  用这些新词再搜，会发现 Round 1 遗漏的公司
  目的: 补全中腰部玩家

Round 3 — 用更边缘的关键词再搜
  从 Round 2 结果中再提取新词
  如果已经没有新的技术路线出现 → 判断收敛
  目的: 确认没有显著遗漏

停止规则: 每轮都是一次"搜索→提取→再搜"的闭环
         新词越来越少 → 自然收敛
         最多搜到 Round 3，不要过度搜索
```

#### 4.2.2 收敛判断标准

```
✅ 条件1: 连续两轮没有发现新的技术路线
   → 只有新公司，但都在已有路线内

✅ 条件2: 新出现的专利都是已有路线的变体
   → 没有全新的材料体系/结构设计/应用方式

✅ 条件3: 主要玩家已覆盖（龙头+中腰部+中国玩家）
   → P0/P1 公司均已发现
```

**定量参考指标：**

| 指标 | Round 2→3 时的收敛阈值 |
|------|----------------------|
| 新发现公司数 / 上轮新发现公司数 | **< 30%** |
| 新发现 IPC 分类号 | **= 0** |
| 新出现的技术概念/术语 | **< 3 个** |

**⚠️ 收敛陷阱：**
- 如果 Round 3 又出现一条全新路线 → 说明搜索策略有盲区，需要调整关键词角度重新搜
- 如果新发现公司远超上轮 30% → 领域可能被严重低估，继续搜
- 如果所有新结果都在验证已有信息 → 确实收敛了，可以输出

#### 4.2.3 实战收敛案例

```
双峰溶剂型丙烯酸酯PSA:
  Round 1: tesa(核心专利), 黑龙江石化院, 斯迪克
  Round 2: 3M(连续挤出), Sekisui(双引发剂)
  Round 3: 无新路线 → 收敛
  → 核心技术集中在tesa一家

锂电池粘结剂(NMP可溶/DMC低溶胀):
  Round 1: 13家, 5条路线
  Round 2: +6家(Samsung SDI, LG Energy, DIC, 巨化...)
  Round 3: +3家(DuPont PEI, Panasonic, 中科院...)
  → Round 3新公司仅占Round 1的23% (<30%), 收敛

折叠屏OCA:
  Round 1: 3M, BOE, Henkel, 华南理工(semi-IPN)
  Round 2: Nitto, 斯迪克(微胶囊), 首尔大学(UV固化/可剥离)
  Round 3: 康得新(量子点), 景华新材(光热双固化)
  → Round 3没有新路线（都是功能型OCA变体）, 收敛
```

#### 4.2.4 特殊情况的处理

| 场景 | 应对 |
|------|------|
| **用户纠正**（"KCI那个是NPWT不是透气敷料"） | 接受纠正，排除误分类，在输出中标注纠正节点，重新搜 |
| **用户补充**（"这个领域最强的是BASF"） | 定向搜该公司的专利+产品资料，刷新公司排名，标注[U]来源（用户提供） |
| **领域完全不熟**（如冠脉支架） | 标注 ★☆☆，更多依赖公开资料+领域知识，明确告知局限性 |
| **领域非常熟悉**（如乳液PSA） | 可快速跳过全景展开，直接进入参数对比 |
| **搜到矛盾信息**（两个源给不同数值） | 优先采信 [P] [T] [C]，标注差异，建议用户自行验证 |

### 4.3 三路搜索执行细则

每轮 Deep Search 都做三路搜索（不是独立的三轮，而是每轮内部的三路并行）：

#### 第一路：公司定向 + 产品资料

```
对每家 P0/P1 公司，做 2 轮搜索:

第1轮: "公司名" + "核心技术关键词" + "patent"
  例: "Arkema" "PVDF" "acrylic copolymer" "binder" "lithium" "patent"

第2轮: VOC Scout 输出的核心 IPC + 公司名
  例: "H01M 4/62" AND "Arkema"

同时搜产品资料:
  例: "Arkema" "Kynar" "HSV" "technical data sheet"
  例: "Solvay" "Solef 5130" "binder" "datasheet"
  例: "BASF" "Acronal V 215" "TDS"
```

#### 第二路：关键词 + IPC 全量扫荡

使用 VOC Scout 输出的 core keywords + core IPC 做无限制搜索，不限公司，抓中小公司：

```
例:
  ("NMP soluble" OR "polyvinylidene") AND ("swelling" OR "electrolyte uptake")
  AND ("lithium") AND "binder" 限定 IPC: H01M 4/62
```

**搜索范围必须覆盖三大数据源：**

| 数据源 | 搜索工具/方法 | 获取内容 |
|--------|-------------|---------|
| **专利文献** | Google Patents, Espacenet, FreePatentsOnline | 权利要求、实施例数据、优先权日 |
| **产品技术资料** | Web搜索 + 公司官网 | TDS, MSDS, 白皮书, 应用指南 |
| **学术论文** | PubMed, ScienceDirect, 知网, Google Scholar | 第三方性能验证、新方向探索 |

#### 第三路：新词提取 → 迭代

从每轮搜索结果中提取四类新词：

```
从专利标题/摘要/产品资料中提取:

① 新关键词（你没用过的术语）
   例: "fibrid", "swelling degree", "electrolyte uptake", "semi-IPN"

② 新公司名（你没听说过的玩家）
   例: "Sichuan Indigo", "Zhejiang Juhua", "Nano Therapeutics"

③ 新 IPC 分类（你没查过的分类号）
   例: "C09J 7/10", "C08L 79/08"

④ 新技术路线（如果出现，说明上一轮不够广）
   例: 有了丙烯酸酯路线→又发现有机硅路线→还要再搜
```

**新词提取后立即进入下一轮搜索。** 不是手动提取——应该用 LLM 做结构化提取（见 §5.3 的 Prompt）。

#### 搜索优先级

当 API 调用次数或时间有限时，按以下优先级分配：

```
1. [最高优先] P0 公司定向 + P0 产品资料        ← 可能直接用现成产品
2. [高优先] 核心 IPC + 核心关键词扫荡           ← 抓到中小公司专利
3. [中优先] P1 公司定向 + P1 产品资料            ← 次选方案
4. [中优先] 新词提取 + 定向搜新词               ← 补盲区
5. [低优先] 前序专利/后续引用（引证滚雪球）     ← 精细分析
6. [低优先] 同族专利搜索                         ← 找不同国家版本
```

### 5.4 专利全文分析架构 — Map-Reduce（逐篇提取 → 汇总对比）

#### 问题

三轮搜索确定了龙头公司+搜索关键词后，用 `patent_search.py` 检索可能返回 50-100+ 篇专利。每篇 `fetch_patent_detail()` 提取的全文约 10,000 字。如果一次性投放 LLM：

```
80 篇 × 10,000 字 = 800,000 字 → 远超 LLM 上下文窗口 → 不可行
```

#### 解法：Map-Reduce

```
Map 阶段（逐篇提取）:
  80 篇专利全文
    → 80 个独立 LLM 调用（每篇 1 次，上下文仅 ~10K 字）
    → 每篇提取结构化 CTQ 记录（~200 字）
    → 80 篇 × 200 字 = 16,000 字结构化数据
    → 可 10 路并行

Reduce 阶段（汇总对比）:
  16,000 字结构化记录
    → 1 个 LLM 调用
    → 输出 CTQ 对比表（去重 + 合并 + 排序）
```

**核心原则：LLM 不读 100 篇专利。LLM 每次只读 1 篇，提取 200 字干货，最后读 80 条的干货汇总。**

#### CTQ 提取 Schema（每篇 ~200 字）

```json
{
  "patent_number": "US10717890B2",
  "assignee": "Arkema France",
  "tech_route": "PVDF改性（共混丙烯酸共聚物）",
  "priority_date": "2016-11-17",
  "ctq": [
    {
      "name": "溶胀率",
      "value": "<30% vol",
      "condition": "EC/DMC/DEC 1:1:1, 60°C, 24h",
      "method": "体积法",
      "unit": "% vol",
      "source": "[P]"
    }
  ],
  "patent_status": "active",
  "expiry_est": "~2038",
  "fto_flag": "check_claims",
  "notes": "丙烯酸含量≤20wt%，非弹性体(Tg>20°C)"
}
```

**规则：**
- 每篇专利提取的 CTQ 记录不得超过 **300 字**
- 如果有多个 CTQ 参数，每条独立记录
- 同公司的多篇专利在 Reduce 阶段自动合并去重
- 优先提取有**数值**的指标，跳过纯定性描述

#### 并发控制

```
10 路并行是安全上限（DeepSeek API 限流）
80 篇 → 10 路并行 → 8 批跑完
每批耗时 ~5-10 秒（LLM 调用 + patent_search 抓全文）
总耗时 ~40-80 秒
```

#### 成本估算

```
单次 Deep Search 的 LLM 调用数:
  初筛: 1 次
  Map: 30-50 次
  Reduce: 1 次
  新词提取: 2-3 次 (R1/R2)
  收敛判断: 1 次
  → 合计 35-56 次 LLM 调用

按 DeepSeek API 定价估算: ~$0.01/次 → ~$0.35-0.56/次 Deep Search
按每天 20 个 VOC: ~$7-11/天 → ~$210-330/月
```

#### Step 0: LLM 初筛（必须实现，不可 deferred）

在逐篇提取之前，必须先用一轮轻量 LLM 从搜索结果中筛选值得读全文的专利：

```
搜索结果 (100篇摘要, ~50K字)
  → LLM 初筛: 哪些专利值得读全文?
  → 输出: Top 30-50 篇的 patent_number 列表
  → 只对这 30-50 篇走 Map-Reduce
  → 节省 50%+ 的全文抓取和 LLM 调用
```

**Pre-screen 输入/输出:**

```
输入: patent_search.py 返回的 Patent 对象列表 (每个含 patent_number, title, assignee, snippet, publication_date)
输出: 按优先级排序的 patent_number 列表 (Top 30-50)

筛选标准:
  1. 优先: 有明确 CTQ 数值指标的（如 "<30% vol", ">3000 g/m²/24h"）
  2. 次优先: 来自 P0 公司的
  3. 次优先: 最近 5 年内的
  4. 排除: 纯方法/设备专利（不涉及材料/产品本身）
  5. 排除: snippet 中没有任何数值指标的
```

**Pre-screen Prompt:**

```text
你是专利分析筛选专家。下面是 {N} 篇专利的摘要信息。
从这些专利中筛选出最值得读全文的 30-50 篇。

筛选标准:
1. 优先选出 Abstract/标题中提到了可量化性能指标的专利
   （如溶胀率、MVTR、铅笔硬度、剥离力等含数字的指标）
2. 优先选出来自行业龙头公司的专利
3. 优先选出近 5 年内的专利
4. 排除纯方法/设备/应用场景类专利（不含材料/产品配方参数）
5. 排除摘要中没有任何数值指标的专利

返回 JSON: {"selected_patent_numbers": ["US123...", "CN456..."], "rationale": "一句话说明筛选逻辑"}
```

#### IPC 搜索适配器

不同搜索引擎对 IPC 的支持格式不同，不能简单把 IPC 塞进 keyword 字符串：

| 搜索引擎 | IPC 参数格式 | 是否原生支持 | 适配方式 |
|---------|-------------|------------|---------|
| **Google Patents** | `"keywords" AND IPC=H01M4/62` | ✅ 支持 URL 中 `&cl=H01M4/62` | 直接拼接在 keyword 字符串中，空格分隔 |
| **USPTO (PatentsView)** | 不支持 IPC keyword 混用 | ⚠️ 部分 | 先按 keyword 搜，再在结果中按 IPC 过滤 |
| **EPO OPS** | CQL: `ic = "H01M4/62"` | ✅ 专用 IPC 参数 | 不能用 keyword 拼接，必须用 CQL 语法 |
| **WIPO Patentscope** | `IC/H01M/4/62` | ✅ URL 格式 | 专用 URL 参数 |
| **中国专利** | `IPC分类号=H01M4/62` | ✅ | 独立参数 |

**规则：** `search_round_company()` 在调用每个搜索引擎时，自行将 IPC 参数转换为该引擎接受的格式。不依赖统一的字符串拼接。

#### Dedup 规则

不同引擎返回的同一件专利不能重复计数。去重按以下优先级：

```
Dedup Key（按优先级）:
  1. patent_number（精确匹配，US10717890B2 ≠ US10717890A1 是不审查的公开号）
     → 优先用授权号（B2/B1），若只有公开号(A1/A2)则用公开号
  2. 同族专利处理: 不合并
     → US10717890B2 和 CN108123456B 算两篇（法律状态不同国家）
     → 但在同一公司的"CTQ汇总"中，如果两篇描述相同CTQ，取较优值
  3. Assignee 名称标准化:
     → "Arkema France" vs "Arkema Inc." → 统一为 "Arkema"
     → 提前维护一份公司名 → 标准名映射表

实现: dedup_key = patent_number.strip().upper().replace(" ", "")
     seen = set()
     for p in all_patents:
         key = dedup_fn(p.patent_number)
         if key not in seen: keep; seen.add(key)
```

#### Map-Reduce 容错与降级

`fetch_patent_detail()` 和 LLM 调用都可能失败。Pipeline 必须追踪成功率：

```
run_map_reduce_pipeline 返回的 stats 必须包含:

  "total_requested": 50,
  "fetch_success": 42,    ← fetch_patent_detail 成功数
  "fetch_failed": 8,      ← 网络超时/503/空文本
  "ctq_extracted": 38,    ← extract_ctq_single 返回了 ≥1 个 CTQ 的数量
  "ctq_empty": 4,         ← 全文拿回来了但没有定量 CTQ（正常，不是每篇都有）
  "fetch_success_rate": "84%"

降级规则:
  fetch_success_rate > 80% → 正常输出，标注 ✅
  fetch_success_rate 60-80% → 输出但标注 ⚠️ "部分专利未能抓取，建议重试"
  fetch_success_rate < 60% → 不执行 Reduce，返回错误 "抓取成功率过低({x}%)，请重试或减少专利数量"

重试策略:
  单篇 fetch 失败 → 不重试（避免触发 API 限流雪崩）
  整体成功率 <60% → 整个 pipeline 重试 1 次（最多 2 次）
  两次 retry 后仍 <60% → 返回已有数据 + 标注失败原因
```
  → 减少 50%+ 的全文抓取量
```

### 5.5 CTQ 参数提取规则

每篇专利/每份产品资料提取以下字段（Map 阶段输出格式）：

| 字段 | 说明 | 提取来源 |
|------|------|---------|
| **公司/申请人** | 专利权人或产品制造商 | 专利首页 / TDS 抬头 |
| **专利号/产品号** | 公开号或授权号，或产品牌号 | 专利首页 / TDS |
| **技术方案** | 核心原理（1-2句话） | 摘要+独立权利要求 |
| **CTQ 参数** | 关键性能指标**数值** | 说明书实施例 / TDS 规格表 |
| **测试条件** | 温度、时间、溶剂配方等 | 说明书实验部分 / TDS 脚注 |
| **测试方法** | 标准号或方法描述 | 说明书实验部分 / TDS |
| **专利状态** | 有效/过期/审查中 | 法律状态查询 |
| **FTO 风险** | 是否构成侵权风险 | 独立权利要求范围 |
| **来源标注** | [P]/[T]/[A]/[C]/[D]/[E] | 见 §3.2 |

**提取规则：**
- 优先提取**有数值的指标**（"溶胀率<30% vol"），而非定性描述（"低溶胀"）
- 记录**测试条件**—同一材料在不同条件下测出的数值差异巨大
- 区分**实施例数据**（最理想情况）和**权利要求范围**（法律保护边界）
- 每个 CTQ 值必须标注 `[来源类型]`

### 4.5 用户反馈闭环

用户纠正是系统最重要的学习信号：

```
用户说: "3M还有个微孔涂布技术你没搜到"
  → 追加搜索: "3M micro-hole coating breathable film"
  → 刷新路线: 新增"微孔涂布"分支
  → 标注 [U] 来源

用户说: "KCI那个是NPWT，MVTR很低"
  → 排除误分类: KCI移到"特殊用途"分支
  → 修正分析: 不再把 NPWT 当成高透气方案
  → 在对话路径中标注纠正节点

用户说: "这个领域最强的是BASF，不是我在Round 1看到的那几家"
  → 重新搜索: "BASF" + "high solids" + "emulsion PSA"
  → 刷新排名: BASF 升为 P0
  → 标注 [U] 来源
```

**系统应记录关键纠正，提升同品类 VOC 的初始准确率。** 尤其是产品-公司映射关系（"Acronal = BASF"）和技术路线误判（"NPWT ≠ 透气敷料"）。
